## Обновление документации

При изменении кода — **всегда** предлагай обновить соответствующую документацию.

### Маппинг изменений

| Что изменилось | Где обновить документацию |
|----------------|---------------------------|
| Бизнес-логика, компоненты, хуки, страницы | `docs/features/` |
| Схема данных, модели, типы, миграции | `docs/architecture/` |
| Инфраструктура, Docker, CI/CD, деплой | `docs/infrastructure/` |

### Как обновлять

1. Определи какие файлы изменились (`git diff --name-only`)
2. По маппингу (см. `config.env`) найди соответствующую документацию
3. Предложи конкретные изменения в формате diff
4. Примени после подтверждения

### Метаданные документов

⚠️ **СТРОГИЙ WHITELIST — НЕ НАРУШАЙ:**

**ТОЛЬКО эти поля** разрешены:

```markdown
> **Status:** current | draft | legacy  
> **Last verified:** YYYY-MM-DD HH:MM  
> **Owner:** @username
```

- **Status:** `current` (актуальный), `draft` (черновик), `legacy` (устаревший)
- **Last verified:** дата и время последней проверки (формат `YYYY-MM-DD HH:MM`)
- **Owner:** ответственный за документ

⛔ **ЗАПРЕЩЕНО добавлять** другие поля:
- ❌ Priority, Tags, Version, Category, Difficulty, etc.
- ❌ Любые поля вне whitelist — **даже если кажутся логичными**

**Если нужно новое поле:**
1. ОСТАНОВИСЬ
2. СПРОСИ пользователя явно
3. НЕ добавляй самостоятельно

При обновлении документа — обнови `Last verified` на текущие дату и время.
